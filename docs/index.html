Overview

Sell the drop. Let the AI answer.

A creator opens a store, links their Instagram, and runs limited-time, limited-quantity drops. Fans land on a public product page — countdown, live stock, one-tap checkout — and an AI assistant answers their questions about the product, so the creator doesn't have to write long copy or sit in a DM inbox. Underneath: multi-tenant isolation, oversell-safe checkout, signature-verified payment webhooks — tested, on CI/CD, running live.

01 — AI customer-service agent

An agent is an LLM plus your own tools plus a loop — hand-built, not a black box.

01

Tool-use loop (Anthropic Claude)

A hand-written loop: send the conversation to the model, it decides which tool to call, we run the Ruby method, feed the result back, repeat until it has an answer. The SDK's higher-level runner hides the loop I wanted to own.

02

Tenant-scoped & read-only

The tools are plain Ruby methods that read this store's data — order status, live stock, return policy — each auto-scoped by acts_as_tenant. No tool can place an order or issue a refund, so the agent is safe by construction. Cross-store isolation is proven by specs.

03

Product-aware

On a drop page the agent is given the product in context, so "is this still available?" resolves without asking which item the buyer means.

04

Off-request, streamed back

A synchronous LLM call would block a web thread for seconds. Instead the request only enqueues a background job; the job runs the loop and broadcasts the reply back to the browser over Action Cable — the placeholder swaps to the answer live.

05

Guardrails

A loop cap stops a misbehaving model from spinning forever; a rescue turns an API timeout into a friendly message instead of a stuck spinner.

02 — Flash-sale drops
01

Influencer identity

Each store links a public Instagram profile — handle, verified badge, avatar; the whole header links out to the real IG page.

02

One product, one page, no cart

A dedicated drop page: original vs. limited price with a discount, remaining stock, and a buy action that goes straight to checkout.

03

Live countdown & states

A Stimulus timer counts down to the sale start or end and reloads at zero; the button is gated on a computed status — upcoming, on sale, sold out, ended — that's a function of the clock and stock, never a stored column.

03 — Commerce core
01

Multi-tenant isolation

Row-level isolation with acts_as_tenant; each store only sees its own data. The admin resolves the tenant from the login, the storefront from the URL slug.

02

Oversell protection

A drop rush is high contention. Checkout decrements stock under a pessimistic lock (SELECT … FOR UPDATE); concurrent buyers can't drive it negative. Covered by a threaded race-condition spec.

03

Payments — Stripe + ECPay

Pluggable: create a pending order → redirect to the chosen gateway → verify the callback's signature (Stripe signature / ECPay CheckMacValue) → only then mark paid. The browser redirect is never trusted; duplicate webhooks are ignored.

04

Order state machine + jobs

AASM drives pending → paid → shipped; on payment a chain of background jobs notifies the store, emails the buyer (Resend HTTP API — the host blocks SMTP), and queues fulfillment.

05

REST API + Postgres-native infra

A JSON API (JWT + rack-attack). Solid Cache and Solid Cable keep cache and Action Cable in Postgres, so the free tier runs with no Redis; images live on Tigris (S3) via Active Storage.

pg_search trigram
No need to run an Elasticsearch cluster at this scale.
Orders
AASM state machine
Explicit, testable states beat a hand-rolled enum + if.
Oversell
Pessimistic lock
Most reliable under high contention; optimistic locking retries a lot.
Payments
Stripe + ECPay, pluggable
One order/webhook flow behind both; the buyer chooses at checkout.
Email
Resend HTTP API
Outbound SMTP is blocked on the host (connections time out).
Images
Active Storage + Tigris
The host has no object storage and an ephemeral disk.
Cache / realtime
Solid Cache + Solid Cable
DB-backed, so no Redis on the free tier.
Frontend
Hotwire + REST
A single stack; no separate SPA for interactions we don't need.
