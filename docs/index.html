Overview

One backend, many storefronts.

Merchants open a store and manage products, inventory and orders; shoppers browse, search and check out on a public storefront. Built around what matters in production — tenant isolation, oversell-safe checkout, an order state machine and signature-verified payment webhooks — all tested, on CI/CD, and running live.

01 — Core features
01

Multi-tenant isolation

Row-level isolation with acts_as_tenant; each store only sees its own data. The admin resolves the tenant from the login, the storefront from the URL slug.

02

Oversell protection

Checkout decrements stock under a pessimistic lock (SELECT … FOR UPDATE); concurrent buyers can't drive it negative. Covered by a race-condition spec.

03

Order state machine

AASM drives pending → paid → shipped; a failed payment rolls back the whole transition, and duplicate webhooks are ignored.

04

Background job chain

On payment: notify the store → email the buyer → queue fulfillment, as a chain of background jobs.

05

Product search

pg_search with Postgres trigram — partial and Chinese matching, no Elasticsearch.

06

Storefront + checkout

A public, login-free storefront to browse, search and order; checkout with a choice of Stripe or ECPay, with an email confirmation.

07

Images + caching

Product images on Tigris (S3-compatible) via Active Storage; storefront cards use fragment caching.

08

Transactional email

Buyer confirmation sent via Resend's HTTP API — the host blocks SMTP, so it goes over the API on port 443.

09

REST API + live dashboard

A JSON API (JWT + rack-attack); the admin order list updates in real time over Turbo Streams + Action Cable.

02 — Checkout

Pluggable payments — Stripe and ECPay behind one order flow.

01

Create order

Create a pending order and a reference, and pick a gateway.

02

Redirect to gateway

Redirect to Stripe Checkout or ECPay; the buyer pays on the gateway.

03

Verify webhook

The gateway calls back; the server verifies the signature (Stripe signature / ECPay CheckMacValue).

04

Mark paid

Only a verified webhook flips the order to paid; duplicate callbacks are ignored.

03 — Technical decisions
Multi-tenancy
acts_as_tenant (row-level)
Apartment's schema-per-tenant needs a migration per schema as stores grow.
Search
pg_search trigram
No need to run an Elasticsearch cluster at this scale.
Orders
AASM state machine
Explicit, testable states beat a hand-rolled enum + if.
Oversell
Pessimistic lock
Most reliable under high contention; optimistic locking retries a lot.
Payments
Stripe + ECPay, pluggable
One order/webhook flow behind both; the buyer chooses at checkout.
Email
Resend HTTP API
Outbound SMTP is blocked on the host (connections time out).
Images
Active Storage + Tigris
The host has no object storage and an ephemeral disk.
Cache / realtime
Solid Cache + Solid Cable
DB-backed, so no Redis on the free tier.
Frontend
Hotwire + REST
A single stack; no separate SPA for interactions we don't need.
